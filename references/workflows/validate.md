# 验证流程

> 验证已有记录的正确性，必要时进行修正

---

## 一、验证类型

| 类型 | 说明 |
|------|------|
| **正确性** | 内容是否仍然有效、准确 |
| **时效性** | 版本/环境变化是否影响适用性 |
| **来源性** | 来源是否权威、是否有更新 |
| **完整性** | 标签是否齐全、格式是否规范 |

---

## 二、状态标记

| 标记 | 含义 |
|------|------|
| ✅ 已验证 | 经实践确认有效 |
| ⚠️ 待验证 | 理论可行未实测 |
| 📘 有效 | 信息准确且当前适用 |
| 🔄 待更新 | 有新版本/变更 |
| 📕 已过时 | 不再适用 |
| ❌ 已废弃 | 错误/有风险 |
| 🔬 实验性 | 非主流方案 |
| 💡 构想中 | 创意阶段 |

---

## 三、验证与修正流程

### 触发场景
| 场景 | 说明 |
|------|------|
| 引用验证 | 引用记录回答时检查适用性 |
| 用户反馈 | 收到"不对"、"过时"反馈 |
| 版本更新 | 技术框架有重大更新 |
| 矛盾检测 | 新记录与已有记录冲突 |
| 新记录写入 | 写入前检查重复和正确性 |

### 修正策略

#### 小幅修正（内容微调）
```markdown
**验证记录**：
- [YYYY-MM-DD] 修正：Unity 2022+ 需要额外配置 xxx
```

#### 重大修正（方案变更）
```markdown
**内容**：
[新方案内容]

<details>
<summary>📜 历史方案（已废弃）</summary>
...
</details>
```

---

## 四、交叉验证

> 不同记录之间的相互佐证

- **知识佐证经验**：记录经验时引用相关知识记录
- **经验验证知识**：用实践结论检验理论知识
- **冲突处理**：
  - 知识更新 → 更新知识记录 + 修正关联经验记录
  - 经验证伪知识 → 检查适用场景或标记过时

### 引用格式
```markdown
### 相关记录
- [相关记录](./other-record.md) - 关联说明
```

---

## 五、冲突处理策略

| 类型 | 策略 |
|------|------|
| **场景差异** | 保留两者，明确适用场景 |
| **版本差异** | 保留两者，标注适用版本 |
| **事实冲突** | 验证后保留正确的，废弃错误的 |

---

## 六、质量检查清单

- [ ] 标签完整？（≥1 领域 + 1 类型）
- [ ] 内容清晰易懂？
- [ ] 代码可运行且无误？
- [ ] 标注了适用版本/环境？
- [ ] 引用了官方/权威来源？
- [ ] 状态标记准确？
- [ ] 文件名自解释？
- [ ] 已更新 INDEX.md？

---

## 七、更新操作

1. **定位记录** → 通过 INDEX.md 或 grep 找到文件
2. **执行更新**：
   - 更新"更新日期"字段
   - 修改状态标记
   - 在"验证记录"添加说明
3. **更新索引** → 如标签或状态变化，同步更新 INDEX.md
4. **推送更改**：`git add . && git commit -m "fix: update [文件名]" && git push`
5. **反馈用户** → 说明更新了什么
