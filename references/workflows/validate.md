# 验证流程

> 验证已有记录的正确性，必要时进行修正

---

## 一、验证类型

### 经验验证
| 类型 | 说明 |
|------|------|
| **正确性** | 确认经验是否仍然有效 |
| **时效性** | 检查版本/环境变化影响 |
| **逻辑性** | 分析推理过程是否闭环 |

### 知识验证
| 类型 | 说明 |
|------|------|
| **来源性** | 确认来源权威性及更新 |
| **准确性** | 核对定义/原理与官方一致 |
| **实践性** | 通过经验检验实际适用性 |

---

## 二、状态标记与可信度

### 经验状态 (5种)
| 标记 | 含义 |
|------|------|
| ✅ **已验证** | 经实践确认有效 (可信度: ⭐⭐⭐⭐⭐) |
| ⚠️ **待验证** | 理论可行未实测 (可信度: ⭐⭐⭐) |
| 🔄 **已过时** | 旧版本适用 (可信度: ⭐⭐) |
| ❌ **已废弃** | 错误/有风险 (可信度: 0) |
| 🔬 **实验性** | 创新/非主流方案 (可信度: ⭐⭐) |

### 知识状态 (3种)
| 标记 | 含义 |
|------|------|
| 📘 **有效** | 准确且适用当前版本 |
| 🔄 **待更新** | 来源有新版本 |
| 📕 **已过时** | 技术环境已淘汰 |

---

## 三、验证与修正流程

### 触发场景
| 场景 | 说明 |
|------|------|
| 引用验证 | 引用记录回答时检查适用性 |
| 用户反馈 | 收到"不对"、"过时"反馈 |
| 版本更新 | 技术框架有重大更新 |
| 矛盾检测 | 新记录与已有记录冲突 |

### 修正策略

#### 小幅修正 (内容微调)
```markdown
**验证记录**：
- [YYYY-MM-DD] 修正：Unity 2022+ 需要额外配置 xxx
```

#### 重大修正 (方案变更)
```markdown
**解决方案**：
[新方案内容]

<details>
<summary>📜 历史方案（已废弃）</summary>
...
</details>
```

## 四、知识与交叉验证

### 知识验证
1. **来源验证**：查阅官方文档/Changelog
2. **准确性**：对比定义与最新规范

### 交叉验证 (知识 ↔ 经验)
> 建立理论与实践的连接
- **知识指导经验**：记录经验时引用相关知识作为理论基础
- **经验验证知识**：记录知识时查找相关经验作为实践证据
- **冲突处理**：
  - 知识源更新 → 更新知识 + 修正关联经验
  - 经验证伪知识 → 检查知识适用场景或标记知识过时

### 关联引用格式
- 经验中：`**理论基础**：[知识点](../knowledge/path.md#anchor)`
- 知识中：`**实践验证**：[相关经验](../experiences/path.md#anchor)`

## 五、冲突处理与质量保证

### 冲突处理策略
| 类型 | 策略 |
|------|------|
| **场景差异** | 保留两者，明确适用场景 |
| **版本差异** | 保留两者，标注适用版本 |
| **事实冲突** | 验证后保留正确的，废弃错误的 |

### 质量检查清单
- [ ] 内容清晰易懂？
- [ ] 代码可运行且无误？
- [ ] 标注了适用版本/环境？
- [ ] 引用了官方/权威来源？
- [ ] 状态/标签准确？

---

## 六、更新操作

1. **定位记录** → 找到需要更新的条目
2. **执行更新**：
   - 更新"更新日期"字段
   - 修改状态标记
   - 在"验证记录"添加说明
3. **推送更改**：`git add . && git commit -m "fix: update [文件名]" && git push`
4. **反馈用户** → 说明更新了什么
